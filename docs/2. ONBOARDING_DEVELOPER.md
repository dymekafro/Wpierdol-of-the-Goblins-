# Onboarding — Wpierdol of the Goblins

Przewodnik dla **początkującego programisty** dołączającego do projektu Unity. Zakładamy, że znasz podstawy programowania, ale Unity może być nowe — wyjaśniamy terminy przy pierwszym użyciu.

**Wersja projektu (stan repo):** Unity **6000.4.7f1**, URP **17.4**, data skanu docs: **2026-05-28**.

---

## 1. Czym jest ta gra?

**Wpierdol of the Goblins** to otwarty **RPG akcji** w jednej dużej mapie — **magicznym, ciemnym lesie**.

- Grasz **druidem** — ostatnim strażnikiem lasu.
- Masz **bazę druida** (Sady) na środku mapy: tam zapisujesz grę, leczysz się i planujesz wyprawę.
- W lesie są **obozy goblinów**. Każdy obóz ma wrogów i **totem** w środku.
- Pętla gry: **baza → eksploracja → walka w obozie → powrót → podbicie obozu** → kolejny obóz.
- **Nie ma leveli** typu „Poziom 1, 2, 3”. Cały świat to jedna scena `World.unity`; trudność rośnie, im dalej od bazy (pierścienie na mapie).

Ton: mroczny las, poważne zagrożenie goblinów, trochę humoru sytuacyjnego w tytule.

Więcej wizji: [GAME_CONCEPT.md](GAME_CONCEPT.md) i [WORLD_DESIGN.md](WORLD_DESIGN.md).

## 3. Jak otworzyć projekt

1. Zainstaluj **Unity Hub** i edytor **6000.4.7f1** (moduł **Windows/Mac/Linux Build Support** jeśli będziesz budować build).
2. W Hub: **Add** → wskaż folder repozytorium (ten z plikiem `ProjectSettings/ProjectVersion.txt`).
3. Otwórz projekt — pierwsze uruchomienie **importuje assety** (może trwać kilka–kilkanaście minut).
4. Po imporcie sprawdź **Console** (okno na dole): powinno być **0 errors** (żółte warningi czasem są OK).
5. Jeśli modele/postacie są **różowe (magenta)** — patrz sekcja „Typowe błędy” (URP).

**Pierwsze uruchomienie gry w edytorze:** otwórz scenę `Assets/_Game/Scenes/MainMenu.unity` i wciśnij **Play** (trójkąt u góry). Nie startuj od `World.unity` — świat jest generowany dopiero po menu i tworzeniu postaci.

## 5. Sceny i flow gry

### Sceny w Build Settings (kolejność ładowania)

| # | Plik na dysku | Nazwa sceny (w kodzie) | Co się dzieje |
|---|---------------|------------------------|---------------|
| 0 | `Assets/_Game/Scenes/MainMenu.unity` | `MainMenu` | Menu: Nowa gra, Kontynuuj, Ustawienia |
| 1 | `Assets/Invector-.../CharacterCreation.unity` | `CharacterCreation` | Rozdysponowanie punktów atrybutów druida |
| 2 | `Assets/_Game/Scenes/World.unity` | `World` | Cały las — generowany w kodzie przy starcie |

> Scena tworzenia postaci **leży w folderze Invector**, ale na obiekcie `Bootstrap` jest skrypt **`CharacterCreationBootstrap`** z naszego projektu — to normalne.

### Przepływ (flow)

```
MainMenu  →  CharacterCreation  →  World
   ↑                                    │
   └──────── Kontynuuj (save) ──────────┘
```

- **Nowa gra:** `GameManager.StartNewGame()` czyści stan i ładuje tworzenie postaci.
- **Kontynuuj:** wczytuje JSON zapisu i od razu ładuje `World` z zapisaną pozycją/HP.
- W `World` obiekt **Bootstrap** ma `WorldBootstrap` — ten skrypt buduje las, gracza, HUD i pauzę.

Stałe nazw scen: `Assets/_Game/Scripts/Core/GameState.cs` → klasa `SceneNames`.

## 7. Jak uruchomić grę

### W edytorze (codzienna praca)

1. Otwórz **`MainMenu.unity`** (`Assets/_Game/Scenes/MainMenu.unity`).
2. Wciśnij **Play**.
3. Kliknij **Nowa gra** → rozdziel punkty atrybutów → **Rozpocznij** → trafiasz do lasu.

**Nie** otwieraj samego `World.unity` i nie dawaj Play — zobaczysz pustą scenę bez pełnego flow (brak menu/atrybutów), choć `WorldBootstrap` częściowo zadziała.

### Sterowanie w świecie (skrót)

| Akcja | Klawisz | Uwaga |
|-------|---------|--------|
| Ruch | WASD | Przez **Invector** (stary Input Manager) |
| Kamera | Mysz | Invector |
| Sprint / skok | Shift / Spacja | Invector |
| Atak wręcz | LPM | `PlayerCombat` (nowy Input System) |
| Fireball | E | Gdy **nie** stoisz przy interakcji |
| Interakcja (obóz, podbicie) | E lub F | Przy ognisku obozu w stanie Cleared |
| Leczenie | Q | Koszt many |
| Pauza | ESC | |

### Kontynuacja zapisu

W menu **Kontynuuj** (aktywne gdy istnieje save). Plik zapisu: ścieżka z logu `[SaveSystem] Zapisano grę: ...` (zwykle folder persistent data użytkownika, plik `wpg_save.json`).

## 10. Git — praca ze znajomym

### Podstawowy workflow

```bash
git pull                    # przed pracą — weź zmiany znajomego
# ... edytujesz w Unity ...
git status                  # co się zmieniło
git add Assets/_Game/ ...   # tylko to co chcesz commitować
git commit -m "Opis zmiany"
git push
```

### Czego NIE commitować

| Folder/plik | Dlaczego |
|-------------|----------|
| **`Library/`** | Cache Unity — gigabajty, generuje się lokalnie (jest w `.gitignore`) |
| **`Temp/`**, **`Logs/`**, **`UserSettings/`** | Lokalne, w `.gitignore` |
| **`Build/`**, **`Builds/`** | Buildy gry — osobno, jeśli w ogóle |

Jeśli Unity pokazuje tysiące zmian w `Library/` — **nie dodawaj ich do commita**.

### Konflikty

- Pliki `.unity` i `.prefab` to YAML — przy konflikcie **dogadajcie się** kto ma rację lub otwórz scenę w Unity i popraw ręcznie.
- Sceny `MainMenu` / `World` mają tylko obiekt **Bootstrap** — większość logiki jest w **C#**, tam konflikty łatwiejsze do mergowania.

### Zasada zespołu

W [TEAM_WORKFLOW.md](TEAM_WORKFLOW.md): **agenci AI nie commitują** — commit robi człowiek po review. U Was we dwójkę: ustalcie kto commituje i kiedy.

## 11. Typowe błędy

### Wszystko różowe (magenta / fiolet)

**Przyczyna:** materiał używa shadera **Built-in**, a projekt jest na **URP**.

**Naprawa:**

1. Menu **WPG → Fix Invector Materials (URP)** (dla paczki Invector).
2. Albo **Window → Rendering → Render Pipeline Converter** → Material Upgrade.
3. Po imporcie nowych modeli: konwersja materiałów do URP Lit.

Więcej: [INVECTOR_INTEGRATION.md](INVECTOR_INTEGRATION.md).

### Play mode nie zapisuje zmian w scenie

Unity **domyślnie cofa** zmiany w scenie po Stop Play. Żeby zostawić: przy Stop wybierz **Yes** (apply to scene) i zapisz scenę **Ctrl+S**.

Zmiany w **skryptach .cs** zapisują się normalnie (niezależnie od Play).

### Gra nie reaguje na WASD po zmianie Input

W **Edit → Project Settings → Player → Active Input Handling** musi być **Both** (oba systemy). Sam **Input System Package** złamie Invector.

Wartość w repo: `activeInputHandler: 2` w `ProjectSettings/ProjectSettings.asset`.

### Przyciski menu nie działają

UI menu działa tylko w **Play mode**, nie w trybie edycji sceny (bez Play).

### Build / Play zła scena startowa

**File → Build Profiles** (Unity 6) — kolejność scen jak w tabeli wyżej; pierwsza = `MainMenu`.

### NullReference w Console po Play

Często brakujący asset (prefab z paczki nie zaimportowany). Sprawdź `[GameAssetRegistry]` i `MISSING` — gra używa **placeholderów** (proste kształty), więc powinna iść dalej.

## 13. Pierwsze zadania (bezpieczne na start)

### A) Zmiana HP goblina szturmowca

1. Otwórz `Assets/_Game/Scripts/Enemies/GoblinStormtrooper.cs`.
2. W metodzie `Awake()` zmień np. `maxHealth = 30` na `maxHealth = 50`.
3. Zapisz, Play z MainMenu, idź do pierwszego obozu (**Pierwsza Zagroda**), sprawdź że goblin żyje dłużej.

### B) Kolor mgły

1. Play z MainMenu → w Hierarchy **Atmosphere** → zmień **Fog Color** / **Fog Density**.
2. Stop → **Yes** → zapisz `World.unity`.

Albo edytuj domyślne wartości w `ForestAtmosphereSettings.cs` (sekcja pól publicznych).

### C) Nowy obóz (zaawansowane)

1. Otwórz `WorldGenerator.cs`, metoda `BuildCamps()`.
2. Skopiuj jeden blok `CampDef` w tablicy, zmień `campId`, `displayName`, `distance`, `angleDeg`.
3. Play — nowy obóz pojawi się na mapie (unikalny `campId`!).

Wzorce ID: [WORLD_DESIGN.md](WORLD_DESIGN.md).

### D) Obrażenia fireballa

1. `PlayerCombat.cs` → `DoFireball()` liczy dmg z `SpellPower * 4`.
2. Albo `PlayerAttributes.cs` → podnieś `intelligence` w bazie druida.
